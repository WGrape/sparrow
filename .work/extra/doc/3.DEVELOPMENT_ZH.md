## 1. 如何上传镜像

### (1) 上传 Basic 镜像

假设本地有一个服务名为 etcd，对应镜像名为 sparrow-basic-etcd，版本为 3.5.0。如果要上传到 Docker Hub，可执行以下命令：

```bash
./sparrowtool upload -t basic -s etcd -v 3.5.0
```

需要注意：上传后远端镜像版本可能与本地版本不同。

- 本地镜像版本：3.5.0
- 远端镜像版本：1.0.{{timestamp}}

![image](https://github.com/WGrape/lexer/assets/35942268/0c8b2850-940b-4d38-b307-e5508b1df9fe)

为了避免误覆盖远端版本，如果要上传与本地同版本的镜像，必须使用 -r 选项（replace），例如 -r true。

```bash
./sparrowtool upload -t basic -s etcd -v 3.5.0 -r true
```

这样远端会生成一个 3.5.0 版本镜像。

![image](https://github.com/WGrape/lexer/assets/35942268/60b3fc54-2535-4319-83d5-1f7b0dc4a37c)

### (2) 上传 App 镜像

用法与上面一致，这里不再重复。

```bash
./sparrowtool upload -t app -s etcd -v latest

./sparrowtool upload -t app -s etcd -v latest -r true
```

## 2. 如何创建新服务

### (1) 使用 Sparrowtool 创建

你可以使用 sparrowtool 创建新服务。比如要创建 jupyter 服务，可以执行：

```bash
# 忘记命令时可先查看帮助
./sparrowtool --help

./sparrowtool new -t service -s jupyter -p 2300 -v 0.1.0
```

执行成功后，项目根目录下会新增一个 /jupyter 目录。

#### ① 分配端口号

如果不知道该用哪个端口，可将新服务（例如 langchain）加入 sparrowtool 文件中 search_env() 的 subdirectories=("etcd" "etcdkeeper" ... "langchain")，然后执行：

```bash
./sparrowtool env -l _port
```

从下图可以看到，系统为 langchain 分配了 [3300, 3400) 端口区间。

<img width="500" alt="image" src="https://github.com/WGrape/sparrow/assets/35942268/ab47592e-2755-4f84-b354-73ef1c6d4b9b">

### (2) 修改服务配置

你需要在 ./jupyter/.env 中至少修改以下镜像变量：

- IMAGE_OFFICIAL_JUPYTER_NAME：官方镜像名。
- IMAGE_OFFICIAL_JUPYTER_VERSION：可选修改，官方镜像版本，例如 IMAGE_OFFICIAL_JUPYTER_VERSION=18.19.0。
- IMAGE_BASIC_JUPYTER_VERSION：可选修改，basic 镜像版本，建议与 IMAGE_OFFICIAL_JUPYTER_VERSION 一致。
- IMAGE_APP_JUPYTER_VERSION：可选修改，app 镜像版本，建议设为 latest。

以上是必须关注的核心变量。若有更多需求，可继续在 ./jupyter 目录中按需调整。

### (3) 在 /.work/config/.env 中加入 ENABLE_SERVICE_LIST

ENABLE_SERVICE_LIST 定义在 /.work/config/.env.amd64 或 /.work/config/.env.arm64 中。你需要把新服务 jupyter 加进去，例如：

```
ENABLE_SERVICE_LIST=("xxx" "xxx" "jupyter")
```

### (4) 删除根目录 /.env

之后建议删除项目根目录下的 /.env 文件，因为系统会自动重新生成新配置。

### (5) 启动服务

现在可以使用以下命令启动：

```bash
./sparrow startone jupyter
```

如果你需要对镜像进行更深度自定义，请继续阅读下一部分：如何更新服务。

## 3、如何更新服务

当镜像内容需要调整以支持新需求时，就需要更新服务，本质上就是更新镜像。

### (1) 更新单个服务

```bash
./sparrow updateone {service}
```

### (2) 使用示例

例如我们需要将 Go 版本从 1.21.1 调整为 1.17.0，可按以下步骤：

1. 先改官方镜像：将 /.env 中 IMAGE_OFFICIAL_GO_VERSION=1.21.1 改为 IMAGE_OFFICIAL_GO_VERSION=1.17.0。
2. 再改 basic 镜像：将 /.env 中 IMAGE_BASIC_GO_VERSION=1.21.1 改为 IMAGE_BASIC_GO_VERSION=1.17.0。
3. 最后更新 go 服务：执行 ./sparrow updateone go。

## 4. 镜像类型与关系

本项目对每个服务采用分层镜像策略。容器最终总是基于 app 镜像启动，basic 和 official 镜像是其上游构建层。

### (1) 镜像类型

- 官方镜像（official image）：来自 Docker Hub 的上游基础镜像，例如 mysql:8.0。
- 基础镜像（basic image）：项目级基础镜像，命名规则为 sparrow-basic-{service}:{version}。
- 应用镜像（app image）：最终运行镜像，命名规则为 sparrow-app-{service}:{version}。

### (2) 命名与版本变量

- IMAGE_OFFICIAL_{SERVICE}_NAME 与 IMAGE_OFFICIAL_{SERVICE}_VERSION：定义上游官方镜像。
- IMAGE_BASIC_{SERVICE}_VERSION：定义 basic 镜像版本。
- IMAGE_APP_{SERVICE}_VERSION：定义 app 镜像版本。

以 MySQL 为例：

- IMAGE_OFFICIAL_MYSQL_NAME=mysql
- IMAGE_OFFICIAL_MYSQL_VERSION=8.0
- IMAGE_BASIC_MYSQL_VERSION=8.0
- IMAGE_APP_MYSQL_VERSION=latest

### (3) 拉取与构建流程

当执行 ./sparrow startone {service} 时，Sparrow 的处理顺序如下：

1. 先尝试从远端拉取 app 镜像。
2. 如果 app 拉取失败，则在本地构建 app 镜像。
3. 构建 app 前，先尝试拉取 basic 镜像。
4. 如果 basic 拉取失败，则基于 official 镜像在本地构建 basic 镜像。
5. 最后通过 docker compose 使用 app 镜像启动容器。

可概括为：

Official image -> Basic image -> App image -> Container runtime

### (4) 为什么拆分 basic 与 app

- 更好利用构建缓存：公共层变化频率更低。
- 提升迭代效率：业务层变更时通常不需要重建全部基础层。
- 职责更清晰：basic 负责运行时基础，app 负责服务级配置和启动。
- 分发更灵活：basic 与 app 可独立上传与复用。

### (5) docker-compose 实际使用哪一层

docker-compose.yml 里的服务镜像直接使用 sparrow-app-*，因此：

- basic 镜像是构建期依赖。
- app 镜像是运行期依赖。

### (6) 开发建议

1. 升级上游运行时版本时，优先同步调整 official 与 basic 版本号。
2. 使用 ./sparrow updateone {service} 完成重建并重启。
3. 本地开发建议 IMAGE_APP_{SERVICE}_VERSION 使用 latest；发布追踪场景再固定版本。
4. 团队协作或 CI/CD 复用时，使用 ./sparrowtool upload 上传镜像。

### (7) 常用命令

```bash
# 启动单服务（优先拉取，失败则本地构建）
./sparrow startone mysql

# 重建并重启单服务
./sparrow updateone mysql

# 上传 basic/app 镜像
./sparrowtool upload -t basic -s mysql -v 8.0
./sparrowtool upload -t app -s mysql -v latest
```

## 5、Nginx 如何反向代理服务

<img width="700" alt="image" src="https://github.com/WGrape/sparrow/assets/35942268/e9ce4bfc-cac7-4474-b1c8-07d17c16cfbe">

### (1) 容器配置

```
GO_HOST_PORT=8002 # 部署 go 服务的宿主机端口
GO_CONTAINER_PORT=8001 # go 容器端口

NGINX_HOST_GO_PROXY_PORT=8004 # nginx 代理 go 服务的宿主机端口
NGINX_CONTAINER_GO_PROXY_PORT=8003 # nginx 代理 go 服务的容器端口
```

### (2) 代理配置

```conf
server {
	listen {{go_proxy_port}};

	location / {
		proxy_pass http://{go_server_addr}:{{go_server_port}};
		proxy_set_header Host $host;
		proxy_set_header X-Real-IP $remote_addr;
		proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
		proxy_set_header X-Forwarded-Proto $scheme;
	}

	error_log /var/log/nginx/nginx_error.log;
	access_log /var/log/nginx/goproxy_access.log;
}
```

## 6、服务在宿主机上的挂载

### (1) 服务目录挂载

每个服务目录都会挂载到容器中的 /home/sparrow/{service} 目录。

### (2) data 目录挂载

每个服务都有一个 data 目录，这是宿主机与容器之间最核心的数据通道。例如：

- MySQL、Redis 等数据库的持久化数据会挂载到各自的 data 目录。
- Python/PHP/Go 等开发环境项目也会挂载到对应的 data 目录。

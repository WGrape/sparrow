## 2. 安装并启动 Docker / Docker-Compose

> Docker 发布说明：https://docs.docker.com/release-notes/

### (1) 安装

#### a. Linux

如果访问 `github.com` 速度慢或失败，可以尝试切换镜像源。

```bash
# 1. 安装 docker
curl -fsSL https://get.docker.com | bash -s docker --mirror Aliyun

# 2. 使用 curl 安装 docker-compose
# 如果速度很慢，建议使用下面的方式。
curl -L https://github.com/docker/compose/releases/download/1.27.2/docker-compose-`uname -s`-`uname -m` > /usr/local/bin/docker-compose
chmod +x /usr/local/bin/docker-compose

# 3. 使用 pip3 安装 docker-compose
pip3 install --upgrade pip
pip3 install docker-compose

# 4. 使用 yum 安装 docker-compose
# 注意版本至少为 1.20 或更高，低版本可能导致 compose 失败。
yum install -y docker-compose
```

#### b. Mac

Mac 版 `Docker Desktop` 安装地址：https://docs.docker.com/desktop/release-notes/#4242 （可选择任意版本）。

### (2) 启动

运行 `docker info` 命令检查 Docker 状态，如果未运行，需要先启动 Docker。

<img width="500" src="https://github.com/WGrape/sparrow/assets/35942268/22c72dca-c62a-43d5-81a1-deb4fca749bd" >

```bash
# 检查 Docker 状态
sudo systemctl status docker

# 启动 Docker
sudo systemctl start docker
```

## 4. 架构不匹配：compose 触发 re-pull 并报 "pull access denied"

### (1) 现象

pull 成功、本地已有 `sparrow-app-{service}:latest` tag，但 `docker compose up` 报错：

```
pull access denied for sparrow-app-{service}, repository does not exist or may require 'docker login'
```

### (2) 根本原因

1. `docker pull` 调用时**没有携带** `--platform`，Docker 拉取与宿主机原生架构匹配的镜像（例如 Intel/amd64 机器上拉到 amd64）。
2. 本地 tag `sparrow-app-{service}:latest` 实际是 **amd64** 镜像。
3. `docker compose up` 读取服务定义中的 `platform: linux/arm64`，发现本地镜像架构不匹配。
4. Compose 自动尝试重新拉取镜像以获取正确架构。
5. `sparrow-app-{service}` 不是真实的 registry 仓库名，拉取被拒绝 —— **报错**。

### (3) 解决方法

`docker pull` 必须始终携带 `--platform ${FROM_PLATFORM}`，这在 `dockerhub.sh` 中已强制执行。如果手动执行 `docker pull`，务必显式加上 `--platform`：

```bash
docker pull --platform linux/arm64 lvsid/sparrow-app-mysql:latest
```

## 5. Docker build 警告：InvalidDefaultArgInFrom 和 SecretsUsedInArgOrEnv

### (1) 现象

执行 `docker build` 时出现以下警告：

```
WARN: InvalidDefaultArgInFrom: Default value for ARG ${IMAGE_OFFICIAL_REDIS_NAME}:${IMAGE_OFFICIAL_REDIS_VERSION}
      results in empty or invalid base image name (line 4)

WARN: InvalidDefaultArgInFrom: Default value for ARG sparrow-basic-redis:${IMAGE_BASIC_REDIS_VERSION}
      results in empty or invalid base image name (line 3)

WARN: SecretsUsedInArgOrEnv: Do not use ARG or ENV instructions for sensitive data (ARG "REDIS_PASSWORD")

WARN: SecretsUsedInArgOrEnv: Do not use ARG or ENV instructions for sensitive data (ARG "MYSQL_ROOT_PASSWORD")

WARN: LegacyKeyValueFormat: "ENV key=value" should be used instead of legacy "ENV key value" format
```

### (2) 原因说明

这些都是 **BuildKit linter 的静态检查警告，不影响构建结果**。

- **`InvalidDefaultArgInFrom`**：`ARG` 声明在 `FROM` 之前且没有设置默认值。BuildKit 在静态分析阶段（`--build-arg` 实际值还未传入时）检查 `FROM` 行，发现变量为空字符串，认为镜像名无效，因此报警告。实际构建时 `--build-arg` 已传入真实值，完全正常。
- **`SecretsUsedInArgOrEnv`**：BuildKit 检测到变量名包含 `PASSWORD`、`SECRET` 等关键字，建议使用 `--secret` 替代 `ARG`/`ENV` 传递敏感数据。这是安全最佳实践提示，不影响功能。
- **`LegacyKeyValueFormat`**：Dockerfile 中使用了旧式的 `ENV key value` 语法，而不是现代的 `ENV key=value` 语法。两种形式功能完全一致，这只是 BuildKit 鼓励使用新格式的风格警告。

### (3) 处理方式

可以安全忽略这些警告，不影响构建和容器运行时。

## 6. docker compose up 时容器一直卡在 "Starting"

### (1) 现象

```
+++++ stage10: do compose... +++++
[+] up 0/1
 ⠼ Container sparrow_container_test_mysql Starting   17.5s
```

容器一直停留在 `Starting` 状态，始终无法变为 healthy。

### (2) 原因

常见原因如下：

- **网络/代理问题**：Docker Compose 在启动容器时会向 registry 请求镜像 digest 做校验。如果代理屏蔽了对 `registry-1.docker.io` 的出站连接，这个请求就会挂起，看起来像容器卡住。这是最常见的原因。

### (3) 解决方法

**针对网络/代理问题**（最常见）：

尝试切换网络，保持良好网络情况，进行多次重试。
